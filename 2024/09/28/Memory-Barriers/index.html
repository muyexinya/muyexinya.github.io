Memory-Barriers

Overview

本文主要介绍Linux内存屏障的概念以及使用方法，目的在于避免过度使用内存屏障导致性能下降或内存屏障力度不够导致稳定性问题。本文中的关于ARM内存类型和内存屏障指令的介绍比较简单，建议阅读arm官方文档Armv8-A memory model和Learn the architecture - Memory Systems, Ordering, and Barriers来学习内存屏障指令的知识。

Arm内存类型

在armv8-A架构中，有两种内存类型：Normal memory 和Device memory。

Normal memory是指RAM, Flash、ROM等可以存放内核、程序等可执行文件的内存，如下图所示：

Device memory是指外设使用的内存，通常情况下，外设寄存器使用 Memory-Mapped I/O（MMIO）映射到内存空间。

Domains

Data memory accesses can take longer and consume more power with cache coherency hardware

than they otherwise would do. This overhead can be minimized by maintaining coherency between

a smaller number of masters while ensuring that they are physically close in the processor. For this

reason, the architecture splits the system into domains, and makes it possible to limit the overhead

to those locations where the coherency is required.

指令乱序与屏障

Reordering

为了提高代码的运行效率，编译器会优化代码并重排指令，使得汇编指令顺序与高级语言逻辑顺序不一致。在弱有序内存架构中，CPU执行指令时，也会使用乱序发射和乱序执行技术来提高CPU并行执行指令的效率,比如多核并发访问共享数据。

乱序执行在下图中，逻辑顺序是：Access1->Access2->Access3，实际指令完成的顺序是Access3->Access2->Access1。

  1. Access1经过Write Buffer将数据写入到外部内存中，
  2. Access2 读指令，cache miss
  3. Access3读指令，cache hit

屏障

在某些场景下，为了避免指令执行的顺序和高级语言的逻辑顺序不一致导致的Bug，我们需要显式使用编译屏障或屏障指令来保证指令的执行顺序与逻辑顺序一致。

编译屏障

barrier()函数可以防止编译器优化导致的指令重排。

ARM内存屏障指令

为了防止CPU的乱序执行对指令重排，我们可以在相关代码中插入内存屏障指令，如Arm指令中的指令同步屏障指令（Instruction Synchronization Barrier,ISB）, 数据同步屏障指令(Data Synchronization Barrier, DSB)和数据存

储屏障指令（Data Memory Barrier, DMB）。

DMB

DMB指令确保了DMB指令前的数据读写指令(LDR, STR)会在DMB指令前执行，并且在DMB之后的数据读写指令之前被指定共享域内的观察者观察到。同时DMB指令可以携带参数限定了哪些类型的显示访问是保序的，以及在指定的共享域中对哪些观察者保序。

Q1. What is an Observer?

A1. An Observer is a processing element or mechanism in the system, such as a peripheral device,

that can generate reads from, or writes to, memory.

Q2. Which DMB qualifier do I use to ensure that two stores that are separated by a DMB are

observed in order by other Observers in the same Inner Shareable domain?

A2. DMB ISHST

Q3. Which memory barrier would be required to ensure that previous memory accesses are

complete before execution continues?

A3. DSB

Q4. What are the architecturally defined Context synchronization events?

A4. The architecturally defined Context synchronization events are as follows:

• Performing an ISB operation

• Taking an exception

• Returning from an exception

• Exit from Debug state
